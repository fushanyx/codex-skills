# Prompt Refiner - 前端开发提示词优化器

## 核心理念

将用户的**原始需求**（可以粗糙、模糊、口语化）转化为**结构化的高质量提示词**，让大模型能一次性准确理解并执行代码修改，减少反复沟通。

## 工作流程

### 第一步：接收原始需求

用户输入原始需求，可能是：
- 口语化描述："把那个登录页面的样式改一下，按钮太大了"
- 模糊需求："优化一下首页性能"
- 带情绪的吐槽："这个表单提交老是报错，烦死了"

不要评判需求质量，直接开始分析。

### 第二步：分析项目上下文

自动扫描项目，获取技术栈和代码约定。按以下优先级读取：

**必读取**（如果存在）：
- `package.json` → 依赖、框架、工具链
- 项目根目录结构 → 目录约定（src/components、src/pages 等）
- `tsconfig.json` 或 `jsconfig.json` → TS/JS 配置
- `.eslintrc*` / `prettier` 配置 → 代码风格

**按需读取**（根据需求类型）：
- 样式需求 → 检查 Tailwind 配置、CSS Modules、styled-components、全局样式文件
- 状态管理需求 → 检查 Redux/Zustand/Pinia/MobX 相关文件
- 组件需求 → 读取目标组件及其目录下的同类组件，分析命名和结构约定
- 路由需求 → 读取路由配置文件

**读取策略**：
- 使用 Glob 和 Read 工具扫描关键文件
- 重点关注已有代码的**命名风格**（驼峰/下划线）、**组件风格**（函数式/类式）、**导入方式**（绝对/相对路径）、**注释语言**
- 分析结果写入提示词的「项目上下文」部分

### 第三步：影响范围分析（边界探测）

**开发者经常不知道改动会影响什么**，这一步由 AI 主动探测，而不是依赖开发者说明边界。

**自动执行以下探测**（对涉及的文件和导出符号）：

1. **谁在用**：用 Grep 搜索目标文件/组件/函数的 import 引用，列出所有调用方
   - 搜索 `import.*from.*['"]目标文件路径['"]`
   - 搜索 `import.*\{.*导出符号名.*\}.*from`
   - 对组件：搜索 JSX 中使用 `<组件名` 的位置

2. **依赖什么**：读取目标文件，列出它 import 的外部依赖（内部模块 + 第三方库）

3. **接口暴露**：检查目标文件的 export，判断修改是否会破坏外部接口
   - Props / TypeScript interface 变更 → 所有使用方可能受影响
   - 函数签名变更 → 所有调用方需要适配
   - 默认导出 → 影响范围较可控
   - barrel file (index.ts) 再导出 → 需追踪到最终消费方

4. **影响等级评估**：
   - 🟢 **低风险**：改动只影响当前文件内部，不改变任何 export 接口
   - 🟡 **中风险**：修改了 export 的 Props/类型/函数签名，有 3-10 个调用方
   - 🔴 **高风险**：修改了公共组件、全局状态、工具函数等，调用方超过 10 个或涉及核心页面

**将分析结果写入提示词的「影响范围」板块**，包含：
- 风险等级和原因
- 具体受影响的文件列表
- 建议的保护措施（如：是否需要逐个检查调用方、是否需要保持向后兼容）

**如果开发者明确说了"我不确定影响范围"或类似问题**，额外执行：
- 用 AskUserQuestion 展示探测结果，让开发者确认哪些文件可以改动、哪些不能碰
- 对于高风险变更，建议分步执行而非一次改完

🔴 **CHECKPOINT**：影响等级为 🟡 中风险或 🔴 高风险时，**必须暂停**——用 AskUserQuestion 展示探测结果（受影响文件列表 + 风险等级 + 保护措施），让用户确认后再进入第四步。低风险可自动通过。

### 第四步：识别需求类型并套用模板

根据需求匹配以下类型，使用对应的优化模板（详见 [prompt-templates.md](prompt-templates.md)）：

| 需求类型 | 关键识别词 | 模板重点 |
|---------|-----------|---------|
| 样式修改 | 样式、颜色、布局、大小、间距 | 引用具体 CSS/样式代码，描述视觉变化 |
| 逻辑修改 | 逻辑、判断、计算、数据处理 | 引用具体函数，描述输入输出变化 |
| 组件开发 | 新建、创建、添加组件/页面 | 参考同类组件结构，描述 Props 和行为 |
| Bug 修复 | 报错、bug、不生效、异常 | 复现步骤 + 期望行为 + 相关代码定位 |
| 重构优化 | 重构、优化、整理、拆分 | 保持行为不变，描述结构变化 |
| 交互增强 | 动画、过渡、交互、体验 | 描述触发条件和效果表现 |

### 第五步：生成结构化提示词

将分析结果组装为以下格式（这是最终输出给用户确认的格式）：

```
## 需求概述
[一句话概括要做什么]

## 当前状态
- **涉及文件**：[具体文件路径列表]
- **当前实现**：[简要描述当前代码做了什么，引用关键代码行]
- **问题/不足**：[当前的问题是什么]

## 影响范围
- **风险等级**：🟢低 / 🟡中 / 🔴高
- **受影响的文件**：[列出所有被探测到的调用方/依赖方]
- **接口变更**：[是否改变 export 接口，如改变列出具体变化]
- **保护措施**：[建议的防御策略，如向后兼容、分步执行等]

## 期望结果
- [具体、可验证的期望行为，用"应该"而不是"最好"]
- [如果有视觉效果变化，描述变化前后的对比]

## 技术约束
- **技术栈**：[从项目分析中提取的技术栈]
- **代码风格**：[从项目分析中提取的风格约定]
- **依赖限制**：[不引入新依赖 / 允许使用已有依赖]

## 不要做的事情
- 不要修改 [具体文件/函数/组件] 的其他部分
- 不要改变 [具体行为/API] 的外部接口（除非「影响范围」中已确认需要）
- 不要引入 [不需要的技术/库/模式]
- 不要重构 [不需要改动的代码]
- 不要碰以下调用方：[列出不能改的依赖方文件]

## 参考代码（如适用）
[贴出需要修改的核心代码片段，标注要改的部分]
```

### 第六步：展示并等待确认

🔴 **CHECKPOINT · 🛑 STOP**：这是执行前的最后一道关卡。输出优化后的提示词，使用 AskUserQuestion 询问用户：

提供三个选项：
1. **确认执行** - 提示词没问题，直接用这个去改代码
2. **需要调整** - 有些地方需要修改（引导用户说明调整点）
3. **补充信息** - 我要补充一些额外的上下文或约束

如果用户选择调整或补充，重新生成并再次确认。

### 第七步：执行代码修改

用户确认后，将优化后的提示词作为完整的任务描述，开始执行代码修改。执行时：

1. 严格按照提示词中的「涉及文件」范围操作
2. 遵循「技术约束」和「代码风格」
3. 时刻检查「不要做的事情」，避免越界
4. 按照「影响范围」中的保护措施行事（高风险时分步执行并逐步确认）
5. 完成后用「期望结果」中的条目做自检

## 特殊处理

### 当用户没有指明具体文件时

主动搜索和推断：
1. 用 Grep/Glob 搜索与需求关键词相关的文件
2. 读取候选文件，判断哪些需要修改
3. 在提示词中明确列出推断结果，让用户确认

### 当需求跨多个文件时

在提示词中：
1. 按修改优先级列出所有文件
2. 说明文件之间的依赖关系
3. 建议修改顺序
4. 对每个文件分别描述要改什么

### 当用户提供了截图/设计稿时

在提示词中：
1. 描述截图中的视觉要素
2. 将视觉目标拆解为具体的 CSS/样式属性
3. 标注哪些是参考图的近似效果，哪些必须精确还原

## 失败模式与降级

以下表格预定义了执行过程中常见的失败场景及处理方式。每种场景给出三段式降级路径：

| 触发条件 | 一线修复 | 仍失败 → 兜底 |
|----------|---------|--------------|
| `package.json` 不存在或读取失败 | 改用 Glob 扫描 `src/` 目录结构推断技术栈 | 在提示词中标注「⚠️ 技术栈未确认，请用户补充」，不猜测 |
| Grep/Glob 搜索目标文件无结果 | 扩大搜索范围：去掉子目录限制、尝试同义词（如 login ↔ signin ↔ auth） | 用 AskUserQuestion 列出候选文件让用户选择，不自动猜文件 |
| Bug 修复场景但用户未提供错误信息 | 主动用 AskUserQuestion 请求：错误消息截图、控制台日志、复现步骤 | 在提示词中标注「⚠️ 缺少运行时诊断信息」，将复现步骤留空待用户填充 |
| 需求类型匹配多个或不属于任何类型 | 分解需求为多个子需求，分别套用最接近的模板 | 使用通用模板（需求概述+当前状态+期望结果+技术约束+不要做），在提示词中标注「⚠️ 混合需求，建议分步执行」 |
| 影响分析探测到 20+ 调用方 | 按调用层级分组（直接依赖 / 间接依赖），只列出直接依赖方 | 标注为 🔴 极高风险，建议用户先确认哪些文件可以改、哪些不能碰，再开始执行 |
| 参考文件（prompt-templates.md / project-analyzer.md）不存在 | 按 SKILL.md 内置模板继续生成，不依赖外部参考 | 在提示词末尾标注「⚠️ 模板参考文件缺失，输出基于内置模板」 |
| 需求涉及非前端代码（后端/数据库/DevOps） | 在提示词中明确标注「超出前端范围」，只生成前端侧的结构化提示 | 告知用户此 skill 专攻前端，建议后端需求单独描述后由其他工具处理 |

**触发规则**：执行第二步到第七步过程中，遇到上表任一触发条件 → 立即走一线修复；一线修复无效 → 走兜底。不要静默跳过失败。

## 执行反例黑名单

以下是执行本 skill 时**必须避免**的反模式。每条来自真实踩坑记录：

| # | 反模式 | 为什么不能做 | 正确做法 |
|---|--------|------------|---------|
| 1 | 跳过影响分析（第三步），因为用户说"很简单不用看" | 开发者常低估改动影响，AI 的探测能力正是此 skill 的核心价值 | 即使用户说简单，仍执行完整影响分析；结果如果确实低风险，只需一句话说明 |
| 2 | 对文件路径做假设而不验证 | 假设 `src/pages/Login.tsx` 存在但实际路径可能是 `src/views/login/index.tsx` | 先用 Glob/Grep 确认文件实际路径，再写入提示词 |
| 3 | 编造 Bug 复现步骤和错误信息 | 运行时信息（报错消息、堆栈）无法从静态代码分析中获得，编造会误导修复方向 | 用 AskUserQuestion 主动请求错误信息；拿不到就在提示词中标注「⚠️ 待补充」 |
| 4 | 简单需求也输出完整 7 段提示词 | "改个颜色"生成含影响分析+技术约束的长文档，用户体验差且浪费时间 | 简单需求（单文件、低风险、无接口变更）使用精简版：需求概述+涉及文件+期望结果+不要做 |
| 5 | 用"适当调整""优化一下"等模糊措辞填写期望结果 | 模糊期望无法验证，等于没写 | 必须给出具体值：颜色 hex、间距 px、函数返回值、行为描述 |
| 6 | 项目分析失败时编造技术栈 | 写错技术栈（如项目用 Vue 却写了 React）会导致整个提示词方向错误 | 分析失败时标注「⚠️ 未确认」，让用户补充后再继续 |
| 7 | 对"不要做的事情"只写泛泛条目 | "不要破坏现有功能"太宽泛，无法约束 AI 的修改行为 | 必须具体到文件、函数、组件名：「不要修改 `useAuth.ts` 中的 `login` 函数签名」 |

**使用方式**：每次执行完提示词生成后，对照本表逐条检查。任一反模式命中 → 立即修正。

## 质量检查清单

生成提示词后，自查以下要点：

- [ ] 涉及的文件路径是否具体、准确（不是"相关文件"这种模糊说法）
- [ ] 期望结果是否可验证（不是"更好看"、"更优雅"这种主观描述）
- [ ] 「不要做的事情」是否覆盖了常见的 AI 过度修改行为
- [ ] 技术约束是否与项目实际技术栈一致
- [ ] 代码风格是否与项目已有风格一致
- [ ] 如果涉及样式，是否给出了具体的值（颜色、间距、字号等）而不是"适当调整"
- [ ] 影响范围分析是否已执行（即使开发者没主动要求）
- [ ] 风险等级评估是否有依据（列出了具体的调用方数量）
- [ ] 高风险变更是否提供了保护措施建议
- [ ] 「不要做的事情」中是否明确了不能碰的调用方文件
