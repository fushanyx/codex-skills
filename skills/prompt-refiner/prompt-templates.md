# 提示词模板库

各场景的完整结构化提示词模板。根据需求类型选择对应模板，填充项目分析结果。

---

## 1. 样式修改模板

**原始需求示例**："把按钮改好看一点"、"间距不太对"

**优化后的提示词模板**：

```
## 需求概述
修改 [组件名称] 的 [具体样式属性] 样式

## 当前状态
- **涉及文件**：[组件文件路径]、[样式文件路径]
- **当前实现**：当前按钮使用 [具体类名/样式方案]，背景色为 [具体值]，
  圆角为 [具体值]，内边距为 [具体值]
  ```css
  /* 当前样式（第XX行） */
  [贴出当前样式代码]
  ```
- **问题**：[具体问题，如"按钮在移动端显示过大"、"与整体设计风格不统一"]

## 期望结果
- 按钮背景色从 [旧值] 改为 [新值]
- 圆角从 [旧值] 改为 [新值]
- 移动端（<768px）按钮高度为 [具体值]，桌面端保持 [具体值]
- 悬浮状态添加 [具体效果]

## 技术约束
- **样式方案**：[Tailwind CSS / CSS Modules / styled-components / 原生CSS]
- **设计规范**：[如果有的话，引用设计 Token]
- **响应式断点**：[项目使用的断点]
- 使用项目已有的 [颜色变量/设计Token]，不新增自定义颜色

## 不要做的事情
- 不要修改按钮的点击事件和业务逻辑
- 不要改变按钮的 DOM 结构和 className
- 不要修改其他组件的样式
- 不要引入新的 CSS 库或工具
```

```

## 当前状态
- **涉及文件**：[文件路径]
- **当前实现**：[函数名] 在第 [行号] 行，当前逻辑为：
  ```[语言]
  // 当前逻辑（第XX-XX行）
  [贴出当前代码]
  ```
- **问题**：[具体问题，如"当输入为空时返回 undefined 而非默认值"]

## 期望结果
- 当 [条件A] 时，应该返回 [具体结果A]
- 当 [条件B] 时，应该返回 [具体结果B]

## 技术约束
- **语言/框架**：[TypeScript / JavaScript + 版本]
- **类型要求**：[函数签名、参数类型、返回类型]
- **错误处理**：[使用 try-catch / 返回错误码 / 抛出异常]

## 不要做的事情

- 不要修改调用此函数的其他文件
- 不要添加不必要的抽象层或工具函数
- 不要修改不相关的其他函数

## 参考代码
[如果逻辑复杂，贴出期望的输入输出示例]
```

- **目标目录**：[组件放置的目录路径]
- **同类参考组件**：[项目中已有的类似组件，用于参考代码风格]
  - [参考组件1路径] - [简要说明]
  - [参考组件2路径] - [简要说明]

## 期望结果
- 组件路径：[完整文件路径]
- 组件名：[PascalCase 组件名]
- Props 接口：
  ```typescript
  interface [组件名]Props {
    [属性1]: [类型]; // [说明]
    [属性2]: [类型]; // [说明]
  }
  ```
- 组件行为：
  - 初始状态应该 [描述]
  - 用户 [操作] 时应该 [响应]
  - [条件] 时应该 [表现]
- 组件应该支持 [无障碍/响应式/主题切换] （如适用）

## 技术约束
- **组件风格**：[函数式组件 + Hooks / 类组件]（参考项目已有组件）
- **状态管理**：[useState / useReducer / Zustand / Redux]
- **样式方案**：[与项目一致的样式方案]
- **导出方式**：[默认导出 / 命名导出]（参考项目约定）
- 参考 [参考组件路径] 的文件结构和命名约定

## 不要做的事情
- 不要使用 [项目中不使用的模式，如 class component / HOC]
- 不要引入新的第三方依赖
- 不要创建额外的工具函数文件（除非必要）
- 不要在组件中直接调用 API（使用项目的请求层）
```

```

修复 [组件/模块名] 的 [问题描述] Bug

## 当前状态
- **涉及文件**：[相关文件路径列表]
- **问题代码**：
  ```[语言]
  // 问题所在（第XX行）
  [贴出有问题的代码]
  ```
- **错误信息**：[完整的错误日志/控制台输出]
- **复现步骤**：
  1. [第一步操作]
  2. [第二步操作]
  3. 出现 [具体错误/异常行为]

## 期望结果
- 执行上述操作时，应该 [正确的行为]
- 不应该出现 [具体的错误表现]
- 边界情况：当 [边界条件] 时，应该 [正确处理方式]

## 技术约束
- **根因分析**：[初步分析的问题原因]
- **修复范围**：[最小修改范围建议]
- 修复后需要确保 [其他相关功能] 不受影响

## 不要做的事情
- 不要重写整个 [组件/模块]，只修复问题部分
- 不要"顺便"优化不相关的代码
- 不要改变 [相关接口/函数] 的外部行为

  ```

## 6. 交互增强模板

**原始需求示例**："加个动画"、"交互体验不好"

**优化后的提示词模板**：

```
## 需求概述
为 [组件/元素名] 添加 [交互类型] 交互效果

## 当前状态
- **涉及文件**：[组件文件]、[样式文件]
- **当前行为**：[当前没有过渡/动画，状态切换是瞬间完成的]

## 期望结果
- **触发条件**：当用户 [悬停/点击/滚动/进入视口] 时
- **效果描述**：
  - [元素] 从 [初始状态] 过渡到 [目标状态]
  - 过渡时长：[300ms]，缓动函数：[ease-in-out]
  - [可选：具体的动画属性，如 transform、opacity]

## 技术约束
- **动画方案**：[CSS transition / CSS animation / Framer Motion / GSAP]
  （优先使用项目已有的方案）
- **性能要求**：只使用 [transform/opacity] 等 GPU 加速属性
- **无障碍**：尊重 prefers-reduced-motion 设置

## 不要做的事情
- 不要使用 JavaScript setInterval/requestAnimationFrame 实现可以用 CSS 完成的效果
- 不要添加不必要的动画库
- 不要对 [不需要动画的元素] 添加过渡效果
- 不要影响页面的首屏渲染性能
```

## 反面案例对照

### 差的提示词 vs 好的提示词

**差**："优化一下这个页面"
**好**："优化 src/pages/ProductList.tsx 的列表渲染性能，当前在数据超过100条时滚动卡顿，
需要实现虚拟滚动（使用已有的 react-virtuoso 依赖），保持现有的筛选和排序功能不变"

**差**："按钮颜色不对"
**好**："src/components/Button.tsx 的 primary 变体背景色应该是 #1677ff（设计稿主色），
当前是默认的蓝色 #1890ff，只改 primary 变体的 backgroundColor，不改其他变体"

**差**："表单提交有问题"
**好**："src/components/LoginForm.tsx 的 handleSubmit 函数在第42行，
当 email 字段包含空格时调 API 报 400 错误。需要在提交前对 email 做 trim() 处理，
只修改 handleSubmit 中的 email 取值，不改变表单结构和其他校验逻辑"
